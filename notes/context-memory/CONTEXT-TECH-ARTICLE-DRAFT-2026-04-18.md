## 1. 为什么要重做 Context：问题不在“能力不够”，而在“结构不清”

在 AMClaw 的早期演进里，Context 系统并不是“没有能力”：  
会话文本可以进 prompt，任务状态可以补充，用户记忆也能注入，系统也能跑起来。

真正的问题出在另一个层面：**上下文系统缺少可解释结构**。  
当系统出现“这轮回复不稳定”或“同样输入结果不同”的时候，我们很难快速回答：

- 本轮 prompt 到底由哪些来源构成？
- 哪些信息被裁剪了？为什么被裁剪？
- 是 session 的问题，memory 的问题，还是预算策略的问题？

这类问题在单次 demo 阶段通常不明显，但进入持续迭代后会迅速放大：

1. 功能越加越多，context 来源越多，拼接逻辑越分散；
2. 口径不统一时，日志、trace、文档会逐渐漂移；
3. 任何改动都可能“感觉更强”，但很难证明是否真的更稳定。

所以我们把目标从“再加能力”转成“先收口结构”：

> 从“能跑”升级到“可解释、可维护、可持续扩展”。

这也是为什么 AMClaw 的 v0.3.3 主线不是继续堆 memory，而是围绕
`SessionState + ContextPack + 可观测` 三件事做收口。

## 2. 选型原则：先状态，再打包；先收口，再扩展

这轮 Context 技术选型，我们采用的是“保守但可持续”的策略，不追求一步到位。

### 2.1 先状态（SessionState），再打包（ContextPack）

为什么先做 SessionState？

- 没有显式状态，系统只能从当轮输入“猜当前意图”；
- 跨轮次目标、阻塞原因、下一步计划会丢失或漂移；
- memory 再强，也不能替代“当前任务状态”的表达。

因此我们先引入最小持久化状态（如 `last_user_intent`、`current_task`、`next_step`、`blocked_reason`），
让系统先回答“用户现在在做什么”。

为什么接着做 ContextPack？

- 有了状态后，还需要统一“怎么喂给模型”；
- 如果仍是分散拼接，定位问题成本依然很高；
- ContextPack 让来源、预算、裁剪理由都变成显式结构。

即：**SessionState 解决“状态来源”，ContextPack 解决“组装方式”**。

### 2.2 先收口语义，再扩展能力

我们刻意不在这一阶段做 embedding、复杂 taxonomy、全量异步重构，原因不是这些不重要，而是：

- 过早扩展会掩盖核心问题（语义口径不统一）；
- 同时引入多个变量，会让收益来源不可解释；
- 没有稳定基线时，复杂机制只会增加维护成本。

因此，本阶段优先级是：

1. 统一语义与入口；
2. 保证可观测和可回归；
3. 再考虑扩展能力。

### 2.3 以“可追溯改动”作为工程约束

每个阶段都要求回答三件事：

1. 为什么做（背景问题）；
2. 做了什么（范围和接线点）；
3. 得到了什么（能力边界变化）。

这也是我们维护 `CONTEXT-TECH-SELECTION-JOURNEY.md` 的原因：  
让技术文章不是事后回忆，而是从开发过程直接生长出来。

## 3. 阶段 0：Context / Memory Minimal（先跑通最小闭环）

在这个阶段，我们做了一个非常克制的选择：  
不追求“最先进的记忆系统”，而是先搭一个**可运行、可验证、可回退**的最小闭环。

### 3.1 为什么先做 Minimal

如果一开始就上复杂机制（例如多路召回、复杂分类、动态预算），会同时引入太多变量：

- 出问题时无法定位是“策略错”还是“数据脏”；
- 任何优化都难判断真实收益；
- 工程复杂度先上去，维护能力跟不上。

所以先把最小链路打通：  
`session_text + current_task + recent_tasks + user_memories`。

### 3.2 这一阶段解决了什么

Minimal 阶段的价值，不是“性能最好”，而是给后续演进提供一个稳定起点：

- 系统具备了最基本的上下文输入能力；
- 有可观测基础（trace/log）可以看行为；
- 有测试基线可以防止后续改动回归。

换句话说，它把项目从“功能拼装”推进到了“系统化迭代”的门槛上。

### 3.3 这一阶段刻意没做什么

- 没做向量检索；
- 没做复杂 memory 分类体系；
- 没做跨层大重构。

这不是技术能力问题，而是节奏控制：  
先把地基打稳，再决定往上盖几层。

---

## 4. 阶段 1：Memory v3 语义收口（统一口径，不堆功能）

Minimal 跑通后，下一步不是“继续加 memory 功能”，而是先把已经存在的 memory 链路讲清楚、做一致。

### 4.1 核心问题：口径不一致

早期最容易出现的混淆是：

- 检索到多少条（retrieved）
- 实际注入多少条（injected）
- 最终命中回写多少条（used）

如果这三层语义不区分，日志和 trace 虽然“看起来有数据”，但无法解释真实行为。

### 4.2 选型：先收口语义，再谈扩展

这个阶段做的不是“更强能力”，而是“更清晰口径”：

1. 统一检索/裁剪/注入/回写的语义边界；
2. 统一日志和 trace 的字段解释；
3. 统一文档和代码中的术语描述。

它的直接收益是：后续改动可以被解释，不再靠感觉争论。

### 4.3 阶段结果

Memory v3 收口后，系统具备了三件关键能力：

- **可理解**：能说清“memory 是怎么进 prompt 的”；
- **可观察**：能看到被裁剪和命中的数量变化；
- **可回归**：有测试保护，不会轻易被后续改坏。

这一点看起来“工程味很重”，但正是后续能稳步推进 SessionState 与 ContextPack 的前提。

---

## 5. 阶段 2：Session Summary 策略（semantic vs truncate）

当 session 文本变长时，Context 系统会面对一个现实问题：  
不压缩会超预算，乱压缩会丢关键信息。

### 5.1 为什么需要双策略

我们没有直接押注单一策略，而是保留了两个策略位：

- `semantic`：尽量保留语义关键段落；
- `truncate`：确定性截断，简单稳定。

这背后的考虑是：不同场景下，两者优劣不一样。  
把策略做成可切换，才能在真实迭代中保持灵活。

### 5.2 关键选型：模块解耦 + 离线评测

这个阶段做了两件很关键的事：

1. 把 summary 逻辑从 `agent_core` 抽到独立模块（降低耦合）；
2. 增加离线评测 CLI（`context_eval`），让策略比较可重复。

重点不是追求复杂评测平台，而是保证“策略调整有依据，且成本足够低”。

### 5.3 阶段结果

Session Summary 阶段带来的收益是结构层面的：

- summary 从“内嵌细节”变成“可复用能力”；
- 策略从“写死逻辑”变成“可比较选项”；
- 后续无论继续优化 semantic，还是调整 truncate，都会更稳。

## 6. 阶段 3：显式 SessionState（C0/C1/C2）

当 summary 和 memory 都有了以后，系统仍然有一个关键短板：  
它知道“用户说了什么”，但不稳定地知道“用户现在处于什么任务状态”。

### 6.1 为什么必须引入 SessionState

如果没有显式状态，系统会长期依赖“从当前输入猜意图”：

- 跨轮次目标容易漂移；
- 阻塞原因容易丢失；
- 下一步动作难以稳定继承。

这类问题不是靠多加 memory 就能解决，因为它本质上是**会话状态管理问题**，不是长期知识召回问题。

### 6.2 接入策略：先最小持久化，再最小接线

我们选择了一个非常小的状态模型：

- `last_user_intent`
- `current_task`
- `next_step`
- `blocked_reason`
- `updated_at`

然后按“低风险顺序”接线：

1. 在存储层增加 `user_session_states`（可读、可 upsert、可清空）；
2. 在 chat 入口读取并注入到 `AgentRunContext`；
3. 在关键时机（如 flush / run 后）做最小回写；
4. 读写失败时只降级记录日志，不阻断主流程。

这保证了系统在“有状态”和“无状态”两条路径上都可控。

### 6.3 阶段结果

C0/C1/C2 完成后，系统获得了一个关键能力：

- 可以显式表达“用户当前在做什么”，而不完全依赖当轮猜测；
- 状态信息进入 trace，可用于后续排障和迭代；
- 退化路径保持稳定（无状态时行为与基线一致）。

这一步看起来“只是加了几个字段”，但实际上把 Context 体系从“文本中心”推进到了“状态+文本并行”。

## 7. 阶段 4：ContextPack 结构化（C3/C4）

完成 SessionState 后，最后一个核心问题是：  
上下文来源虽然更丰富了，但组装逻辑仍分散在多处，难统一解释。

### 7.1 为什么要引入 ContextPack

如果没有统一打包层，系统会持续面临三个工程问题：

1. 入口分散：很难定位“这轮 prompt 到底怎么组出来的”；
2. 裁剪不透明：很难解释“为什么某段信息被丢弃”；
3. 演进成本高：每次新增上下文来源都容易改出旁路。

ContextPack 的价值就是把这些问题变成结构化对象来处理。

### 7.2 接入方式：统一入口 + 可观测字段

我们把主链路明确收敛为：

`build_context_pack -> render_prompt_from_context_pack`

并把 pack 级可观测同步进 trace/log：

- `context_pack_present`
- `context_pack_section_count`
- `context_pack_total_chars`
- `context_pack_drop_reasons`
- `context_pack_built` / `context_pack_trimmed` 事件

这样，系统不只是在“生成 prompt”，而是在“生成可解释的 prompt”。

### 7.3 阶段结果

C3/C4 完成后，Context 系统出现了质变：

- 组装入口统一，避免继续散落拼接；
- 裁剪理由可追溯，行为解释成本显著下降；
- 新增上下文来源时有固定扩展位（section kind + policy）。

换句话说，AMClaw 的 Context 从“能工作”进入了“可持续工程化”阶段。

## 8. 为什么暂时不做：embedding、复杂 taxonomy、全量异步化

很多团队在 Context 演进阶段会本能地走向“更复杂”：  
向量召回、复杂 memory taxonomy、跨层全量重构。

这些方向并不错误，但在 AMClaw 当前阶段，我们刻意选择暂缓，核心原因有三点：

### 8.1 复杂机制会放大口径不一致问题

如果基础语义和观测口径还不稳定，引入更复杂机制只会让系统更难解释。  
工程上最危险的状态不是“功能少”，而是“功能多但说不清”。

### 8.2 同时引入多个变量会让收益来源不可归因

当你同时改了召回、状态、预算、渲染入口，任何结果变化都无法判断主因。  
这会让团队回到“凭感觉选型”的状态。

### 8.3 当前瓶颈是结构稳定性，不是能力上限

在现阶段，系统的主要痛点是：

- 状态表达是否稳定；
- 上下文组装是否可解释；
- 裁剪是否可追溯。

这些问题不解决，再强的召回机制也会被系统噪声抵消。

因此我们明确把“先收口，再扩展”作为阶段策略：  
先把 SessionState + ContextPack 跑稳，再考虑下一层能力。

## 9. 结果与边界：我们得到了什么，还没得到什么

截至 C3/C4，AMClaw 的 Context 系统已经从“最小可用”进入“可持续迭代”阶段，但它仍然有明确边界。

### 9.1 已获得的能力

1. **状态可表达**：`SessionState` 让跨轮次意图与阻塞信息可持续存在。
2. **组装可统一**：`ContextPack` 提供了单入口组装，不再依赖多处字符串拼接。
3. **裁剪可解释**：trace/log 能看见 section 数量、总量和 drop reason。
4. **演进可追踪**：技术选型和接入过程已形成连续文档链路。

### 9.2 仍然存在的边界

1. 仍是“规则+结构化”主导，不是“智能检索”主导。
2. 状态回写仍偏最小，不做深层任务语义自动推导。
3. 预算策略仍偏静态，缺少更细粒度的动态调度。
4. trace 可观测增强了，但还未形成自动化分析流水线。

### 9.3 风险提示

如果下一阶段没有保持“单变量推进”的纪律，系统很容易出现：

- 解释成本重新上升；
- 策略回归难以及时定位；
- 文档与实现再次漂移。

所以这阶段的真正成果不是“多了几个字段”，而是建立了一个可持续演进的工程方法。

---

## 10. 下一步：C5 稳定化与参数治理

在 C3/C4 完成后，最合适的下一步不是继续加功能，而是做一次“稳定化冲刺”。

### 10.1 目标

把 ContextPack 的策略从“代码常量”推进到“可控参数”，同时保持语义不漂移。

### 10.2 建议执行顺序

1. **参数化 section policy（最小）**
   - 将关键 `priority/max_chars/pinned_lines` 提升为可配置（先 TOML）。
2. **固化两条回归场景**
   - 有 state + 有 memory；
   - 无 state + 无 memory（降级路径）。
3. **补轻量 trace 摘要脚本**
   - 从当日 trace 提取 section 分布与 drop reason 统计。
4. **文档持续更新**
   - 每个小迭代都追加到 `CONTEXT-TECH-SELECTION-JOURNEY.md`。

### 10.3 C5 完成标准（建议）

- 策略参数可调，但默认行为与当前基线一致；
- 关键回归场景稳定通过；
- 能快速回答“参数变化带来了哪些上下文分布变化”。

---

## 结语

AMClaw 这轮 Context 演进的核心，不是“做了更复杂的 AI 能力”，而是：

> 先把系统变得可解释，再让它变得更强。

从 SessionState 到 ContextPack，这条路径的价值在于它建立了工程秩序：  
每一步都有动机、边界、实现和结果，系统可以在可控前提下持续前进。
