# NEXT-STEPS

当前这份文件只记录“接下来最值得做什么”，不重复描述已经完成的能力。当前真实状态请看 `PLAN.md`。

## 当前优先级

### 1. 把 Agent Trace 接到真实聊天链路

- 让真实微信消息触发的 Agent 运行也能产生日志
- 在 trace 中补 `user_id`、`message_id`、来源类型等上下文
- 让复盘时能从“哪条消息”直接追到“模型怎么做、工具怎么调、最终回了什么”

### 2. 完善 Trace 浏览入口

- 在现有 `index.jsonl` 之外，增加按日可读的 `index.md`
- 让每天的 run 可以先快速总览，再决定要不要点进单次 JSON / Markdown

### 3. 系统级日志与错误结构

- 把关键链路从 `println!` / `eprintln!` 逐步收敛到统一日志体系
- 覆盖 `chat_adapter`、`pipeline`、`task_store`
- 让登录、任务消费、浏览器抓取、人工补录的错误更容易检索与归因

### 4. 内容处理继续深化

- 提升通用网页正文抽取质量
- 在公众号之外逐步补分类与摘要链路
- 让 pipeline 状态比现在更明确
- 对“普通短链最终跳到公众号”的场景，当前先保持观测，不立即自动升级到 browser capture
- 等积累更多真实样本后，再决定是否引入二阶段抓取策略

### 5. 汇总与调度

- 为后续 `scheduler` / `reporter` 留稳定接口
- 评估日报生成与回传的最小闭环

## 当前明确暂不做

- `tokio` 全量迁移
- `sqlx` async 化
- 模式策略的大改版
- 多聊天应用接入层重构

## 下一轮建议顺序

1. 先把 Agent Trace 接到真实聊天链路
2. 再补每天的 `index.md`
3. 然后做系统级结构化日志
4. 再补通用网页抽取、分类、摘要
5. 最后再做调度、日报与更大架构迁移

## 开工标准

下一轮继续开发前，至少保证：

- 现状文档与仓库行为一致
- 真实 Bot 登录与消息链路仍可用
- 公众号浏览器抓取链路不回退
- Agent Trace 当前 JSON / Markdown / `index.jsonl` 生成逻辑不回退
- `retry`、`状态`、`最近任务`、`待补录任务` 这些命令的行为与文档一致
