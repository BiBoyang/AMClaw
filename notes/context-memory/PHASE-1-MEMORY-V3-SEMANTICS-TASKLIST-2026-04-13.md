# Phase 1 任务清单：收口 Memory v3 语义（2026-04-13）

## 目的

本清单用于把 `v0.3.3` 路线中的 **Phase 1：收口 `Memory v3` 语义** 拆成可执行任务，避免后续推进时只停留在概念层。

本阶段的目标不是增加新 memory 能力，而是把当前已经存在的 memory 链路在以下四个方面收口：

1. 写入语义一致；
2. 检索 / 注入 / 命中统计一致；
3. 文档 / 日志 / Trace 口径一致；
4. 为后续 `session state` 与 `context pack` 奠定干净基础。

---

## 一句话目标

> 把当前 `Memory v3` 从“能工作”收口成“语义清晰、统计一致、可继续扩展”的状态。

---

## 当前已知问题

基于当前代码与文档，对 Phase 1 最值得优先处理的问题整理如下：

### 1. 自动记忆与显式记忆的写入语义未完全一致

设计稿中，自动提炼的记忆应当是：

- `memory_type='auto'`
- `priority=60`

但当前聊天入口自动提炼后，实际仍走通用显式写入路径，容易让：

- 自动记忆伪装成显式记忆；
- 检索排序失真；
- 后续扩 memory 类型时语义继续混乱。

### 2. “命中”语义还不够严格

当前系统已有：

- 检索
- 注入
- 命中回写

但“命中”在语义上仍偏模糊：

- 是“被检索到了”？
- 是“被注入 prompt 了”？
- 还是“真的帮助了决策”？

如果这些定义不拆清，后续 trace 和评测会越来越难解释。

### 3. `use_count` 的语义需要固定

当前实现里，`use_count` 更接近“被取到/被注入次数”，但文档语言容易让人理解成“真正有用次数”。

必须在 Phase 1 明确：

- 它到底是 `retrieved_count`
- 还是 `injected_count`
- 还是 `accepted/useful_count`

一旦确定，就要统一：

- 字段命名
- 日志说明
- Trace 描述
- 文档表述

### 4. Trace 与日志统计口径还可以更细

当前已经有：

- `memory_hit_count`
- `memory_total_chars`
- `memory_ids`

但后续真正做评测时，更有价值的是区分：

- `retrieved_memory_count`
- `injected_memory_count`
- `retrieved_memory_total_chars`
- `injected_memory_total_chars`

本阶段未必都要全部自动判定“useful”，但至少应该先把“检索”和“注入”拆开。

---

## Phase 1 总体原则

### 原则 1：只收口语义，不扩大战线

本阶段不引入：

- 向量库
- 复杂 taxonomy
- 复杂 rerank
- 新一轮自动提炼体系

### 原则 2：先统一定义，再动实现

先确定：

- 什么叫 `auto`
- 什么叫 `explicit`
- 什么叫 `retrieved`
- 什么叫 `injected`
- `use_count` 代表什么

然后再改代码。

### 原则 3：统计字段必须可解释

任何新增统计都必须回答：

- 这个值在哪个时刻产生？
- 它统计的是“看到了”、还是“送进去了”、还是“真的有用”？

### 原则 4：不破坏当前最小可用链路

Phase 1 的所有改动都应保持：

- 无记忆时可退化；
- 检索失败不阻断主链路；
- 当前行为尽量保持兼容。

## 建议拆分顺序

推荐把 Phase 1 拆成 4 个子任务推进。

---

## 子任务 A：统一 Memory 语义字典

### 目标

先把所有关键术语定义清楚。

### 需要明确的定义

#### 1. 写入来源

- `explicit`
  - 用户明确要求系统记住的内容
- `auto`
  - 系统从聊天中自动提炼出的内容

#### 2. 统计阶段

- `retrieved`
  - 从存储中被检索出来
- `injected`
  - 被放入本轮 prompt / context
- `useful`
  - 真正帮助本轮决策

> 当前阶段可以先不自动判断 `useful`，但必须明确它和 `retrieved / injected` 不是一回事。

#### 3. `use_count`

必须明确它在本项目中代表哪一个：

- 方案 A：被检索次数
- 方案 B：被注入次数

本阶段建议优先选 **方案 B：被注入次数**，因为它比“被检索到”更接近实际消费。

### 输出物

需要同步更新：

- `notes/context-memory/STATE-MANAGEMENT-AND-CONTEXT-MEMORY-PLAN-2026-04-12.md`
- `notes/context-memory/CONTEXT-MEMORY-EVOLUTION-ROADMAP-2026-04-13.md`
- `PLAN.md`
- `NEXT-STEPS.md`

### 完成标准

1. 术语定义有固定文本；
2. 文档之间不再出现相互冲突的说法；
3. 后续实现改动都能引用同一套定义。

---

## 子任务 B：修正自动记忆写入语义

### 目标

保证自动提炼的记忆不会再落成显式记忆。

### 需要修改的方向

重点检查：

- `chat_adapter`
- `task_store`
- 自动记忆写入调用链

### 预期结果

自动提炼写入时应满足：

- `memory_type='auto'`
- `priority=60`

显式写入仍保持：

- `memory_type='explicit'`
- `priority=100`

### 需要补的验证

至少需要有回归覆盖：

1. 显式写入后类型正确；
2. 自动提炼写入后类型正确；
3. 显式记忆排序仍高于自动记忆；
4. 自动记忆不会污染“显式偏好优先”的语义。

### 完成标准

1. 自动记忆不再伪装成显式记忆；
2. 检索排序与设计稿一致；
3. 回归测试覆盖该语义。

---

## 子任务 C：拆分检索与注入统计

### 目标

把当前较粗的 `memory_hit_count` 拆成更可解释的统计。

### 建议新增或收口的统计口径

至少区分：

- `retrieved_memory_count`
- `retrieved_memory_total_chars`
- `injected_memory_count`
- `injected_memory_total_chars`

如果短期内不想增加太多字段，也至少要做到：

- 明确 `memory_hit_count` 指的是哪一个阶段；
- Trace / 日志 / 文档统一采用同一含义。

### 推荐实现策略

#### 最小做法

不一定马上重命名字段，但要：

1. 在代码注释和文档里明确当前 `memory_hit_count` 的真实含义；
2. 如有精力，再新增 `retrieved_` / `injected_` 字段；
3. 保持向后兼容，不急着一次性删旧字段。

#### 更优做法

trace 里同时保留：

- 当前兼容字段
- 新的更精确字段

等系统稳定后再考虑清理旧字段。

### 需要覆盖的点

1. 检索成功但未注入时，统计是否正确；
2. 注入预算裁剪后，`retrieved` 与 `injected` 是否可能不同；
3. Trace / 日志展示是否一致。

### 完成标准

1. memory 统计从“模糊命中”变成“阶段明确”；
2. trace 与日志更适合后续评测；
3. 不影响当前功能链路。

---

## 子任务 D：补 Phase 1 回归测试与收口检查

### 目标

让 Phase 1 不是只改语义，而是能被验证。

### 必测项

#### 1. 写入语义

- 显式写入得到 `explicit`
- 自动写入得到 `auto`
- 不同优先级排序符合预期

#### 2. 检索与预算

- `status='suppressed'` 不参与结果
- 去重仍有效
- 单条长度预算仍有效
- 总预算仍有效

#### 3. 统计与 Trace

- memory 统计字段存在且语义正确
- Trace 落盘包含更新后的统计字段
- 无记忆场景仍正常

#### 4. 回退行为

- memory 检索失败时不阻断主链路
- 标记已使用失败时不阻断主链路

### 收口检查表

Phase 1 完成前，应逐项确认：

1. 文档定义是否统一；
2. 自动记忆写入是否已修正；
3. 统计口径是否统一；
4. Trace 是否仍可读、可解释；
5. 当前行为是否兼容；
6. 回归测试是否覆盖新增语义。

---

## 推荐执行顺序（开发视角）

按实际开发建议这样推进：

1. **先改文档定义**
   - 避免边写代码边改概念
2. **再修自动记忆写入**
   - 这是当前最明确的设计 / 实现偏差
3. **再拆统计字段或统一统计语义**
   - 给后续 trace 评测打底
4. **最后补测试与验收**

---

## 明确不在 Phase 1 做的事

本阶段不处理：

- `session state` 结构化
- `context pack` 抽象
- `project_fact / lesson` 新类型扩展
- 向量检索
- 复杂 rerank
- 复杂自动提炼

这些都属于后续 Phase。
