# AMCLAW-FEATURE-ROADMAP

更新于 2026-04-10。

这份文档回答的不是“当前已经完成什么”，而是：

1. `AMClaw` 在“收集文章与笔记”之外，后续最值得长出哪些能力
2. 这些能力的优先级怎么排
3. 这些能力更适合放在 `core`、`service`、`skill` 还是 `product` 层

这份文档和其他文档的关系如下：

- `README.md`：当前能跑什么
- `PLAN.md`：当前真实状态
- `NEXT-STEPS.md`：下一轮先做什么
- `CORE-SKILL-ROADMAP.md`：整体分层路线
- `AGENT-RUNTIME-IMPLEMENTATION-PLAN.md`：runtime 实施计划
- 本文档：**未来功能路线**

## 一、产品方向重新定义

`AMClaw` 不应长期停留在“收藏文章 / 收藏笔记”的单一定位上。

更合适的目标是：

- 一个以微信为入口的 **个人知识与任务助理**

长期看，`AMClaw` 应围绕四条主线成长：

1. `Capture`
   - 收文章
   - 收笔记
   - 收想法
   - 收待办
2. `Understand`
   - 摘要
   - 分类
   - 打标签
   - 建立主题关系
3. `Act`
   - 定时提醒
   - 任务推进
   - 执行计划
   - 失败回执
4. `Review`
   - 日报 / 周报
   - 历史检索
   - 知识回顾
   - 阶段总结

## 二、功能分层标注规则

为避免后续再次把“功能本身”和“实现承载方式”混在一起，本文档中每项功能都带一个 `推荐归属`。

可选值：

- `core`
  - runtime 能力、状态框架、安全、调度框架等底座
- `service`
  - 稳定、确定、可测试、带副作用或持久化的执行能力
- `skill`
  - 可替换的策略、提示词、工作流编排
- `product`
  - 直接面向用户的入口、界面、交互体验

说明：

- 某个功能在短期和长期可能分属不同层
- 如果会变化，本文会明确写“短期归属”和“长期归属”

## 三、P0：最值得优先发展的功能

这些功能不是“以后也许会做”，而是最值得让 `AMClaw` 尽快长出来的能力。

### 1. 定时任务与日报 / 周报

- `目标`
  - 让 `AMClaw` 从“被动收集”变成“主动服务”
- `可见能力`
  - 每日新增收藏摘要
  - 待处理任务清单
  - 待补录任务提醒
  - 每周主题汇总
- `短期推荐归属`
  - `core`：调度触发框架
  - `service`：日报数据聚合
  - `skill`：日报生成策略
  - `product`：微信回传与后续 Web 展示
- `为什么优先`
  - 当前已经有任务、归档、状态，离“主动推送”只差调度与汇总
- `当前进展`
  - 本地 Markdown 日报已落地
  - 微信内可通过 `日报` / `今日整理` 查询当天摘要
  - 最小自动微信推送已落地，依赖 `report_to_user_id` 与已持久化 `context_token`

### 2. 会话恢复 / Session Resume

- `目标`
  - 让会话跨消息、跨中断继续推进
- `可见能力`
  - “上次做到哪了”可以接着做
  - 长任务不会因为一条新消息就完全丢失上下文
- `短期推荐归属`
  - `core`
- `长期可延伸`
  - 可与 `skill` 结合决定如何恢复上下文
- `为什么优先`
  - `AMClaw` 正在从单回合 Agent 往多步 runtime 演进，这一步价值非常大
- `当前进展`
  - 已补最小会话持久化与进程重启恢复
  - 仍未形成完整的长期会话状态机

### 3. 持久记忆系统

- `目标`
  - 让 `AMClaw` 不只记“这次消息”，还记“长期偏好和长期主题”
- `可见能力`
  - 用户偏好记忆
  - 最近关注主题
  - 常见来源偏好
  - 对话 / 文章长期上下文
- `短期推荐归属`
  - `core`：memory 框架
  - `service`：持久化与索引
  - `skill`：记忆读写策略
- `为什么优先`
  - 这会让 `AMClaw` 从收藏器变成真正的个人知识助理
- `当前进展`
  - Memory v1/v2 已落地：支持显式写入用户记忆，并支持最小主题 / 偏好自动提炼
  - 仍未实现长期主题记忆与语义检索

### 4. Plan & Execute

- `目标`
  - 让 `AMClaw` 不只会回答，还能“分解任务并逐步执行”
- `可见能力`
  - 生成 todo plan
  - 按步骤执行
  - 跟踪执行进度
- `短期推荐归属`
  - `core`
- `长期可延伸`
  - 某些任务规划策略可 skill 化
- `为什么优先`
  - 当前最小 ReAct runtime 已经落地，这是下一步自然延伸

### 5. 本地 Web 控制台

- `目标`
  - 让 `AMClaw` 有一个更适合管理任务、归档和 trace 的界面
- `可见能力`
  - 浏览最近任务
  - 浏览待补录任务
  - 浏览已归档文章
  - 浏览 agent trace
  - 人工修订内容
- `短期推荐归属`
  - `product`
- `长期可延伸`
  - 可作为 skill / memory / trace 的可视化入口
- `为什么优先`
  - 现在大量信息都在 SQLite 与本地文件里，缺少统一浏览面板

### 6. 长消息分段与处理中回执

- `目标`
  - 提升微信场景下的真实可用性
- `可见能力`
  - 长回复自动拆段
  - 长任务中途提示“处理中”
  - 多步执行时回执进度
- `短期推荐归属`
  - `product`
- `长期可延伸`
  - 可与 plan / execute 深度结合
- `为什么优先`
  - 成本低、体验提升明显

## 四、P1：很值得做，但可排在 P0 之后

### 7. Web Search / 链接背景补检索

- `目标`
  - 不只归档用户发来的链接，还能主动补齐背景
- `可见能力`
  - 补项目主页
  - 补 GitHub 仓库
  - 补相关新闻 / 讨论
  - 补同主题资料
- `短期推荐归属`
  - `service`
- `长期可延伸`
  - 搜索策略与筛选策略可 skill 化
- `为什么值得做`
  - 能把“收藏”升级成“调研入口”

### 8. Mention catch-up / 会话补读

- `目标`
  - 在群聊或连续对话里，更好地理解最近上下文
- `可见能力`
  - 被 @ 时补读最近消息
  - 理解上一轮上下文，而不是只看最后一句
- `短期推荐归属`
  - `core`
- `长期可延伸`
  - 群聊处理策略可 skill 化
- `为什么值得做`
  - 现在有 session merge，但未来需要更完整的上下文补齐

### 9. Provider 抽象层继续统一

- `目标`
  - 让后续 tool use、memory、skills 都不被单一模型 provider 绑定
- `短期推荐归属`
  - `core`
- `为什么值得做`
  - 当前已有多 provider 能力，但后续需要继续统一 structured output / tool use 契约

### 10. MCP 接入

- `目标`
  - 给 `AMClaw` 一个比“全部内建”更灵活的扩展面
- `短期推荐归属`
  - `service`
- `长期可延伸`
  - tool / skill 层都可从中获益
- `为什么值得做`
  - 适合未来接浏览器、搜索、桌面、日历、外部知识库等能力

### 11. 归档内容二次加工

- `目标`
  - 不只归档正文，还能进一步形成知识材料
- `可见能力`
  - 标签
  - 主题聚类
  - 摘要
  - 要点提取
  - 相关收藏关联
- `短期推荐归属`
  - `service`
- `长期可延伸`
  - 分类 / 摘要 / 聚类策略适合 skill 化

## 五、P2：中长期方向

### 12. 多渠道接入

- `目标`
  - 一套 runtime，多种聊天入口
- `候选渠道`
  - Telegram
  - Discord
  - Feishu / Lark
  - Web
- `短期推荐归属`
  - `product`
- `当前建议`
  - 先预留 adapter 结构，不急着立刻接多渠道

### 13. Sub-agent / Agent Swarms

- `目标`
  - 复杂任务拆分给多个 agent 协作
- `短期推荐归属`
  - `core`
- `当前建议`
  - 等单 agent runtime 稳定后再做

### 14. 容器隔离 / 凭证安全代理

- `目标`
  - 提升未来强工具调用能力下的安全边界
- `短期推荐归属`
  - `core`
- `当前建议`
  - 先在路线图中保留，不作为当前短期首要任务

### 15. Skill 系统继续落地

- `目标`
  - 把上层策略型能力变成可替换的 skill
- `适合 skill 化的内容`
  - 聊天回复风格
  - 文章摘要策略
  - 文章分类策略
  - 日报生成策略
  - 任务失败解释
- `短期推荐归属`
  - `skill`
- `为什么不是当前第一优先级`
  - 还需要先把 runtime / tool / context / memory 底座继续做稳

## 六、建议的实际开发顺序

1. 定时任务与日报 / 周报
2. 会话恢复
3. 持久记忆系统
4. 本地 Web 控制台
5. Plan & Execute
6. Web Search
7. MCP
8. 多渠道接入
9. Sub-agent
10. 更完整的 skill 系统

## 七、功能与分层的配套理解

这份路线图和“skill 化”不是冲突关系。

正确理解是：

- 功能路线图回答“以后要做什么”
- 分层路线图回答“这些功能应该放在哪层”

例如：

- “定时任务”本身是一个功能目标
  - 调度框架属于 `core`
  - 汇总数据属于 `service`
  - 日报生成策略适合 `skill`
  - 微信展示和 Web 展示属于 `product`

所以后续不要把“值不值得做”和“该不该 skill 化”混成一个问题。

## 八、当前结论

当前 `AMClaw` 的未来功能方向，应重点从下面这些里选：

- 定时任务 / 日报
- 会话恢复
- 持久记忆
- Plan & Execute
- Web 控制台
- Web Search
- MCP

一句话总结：

- `AMClaw` 的未来不是“多收几种文章”
- 而是逐步成为 **个人知识与任务助理**
